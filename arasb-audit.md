# گزارش جامع تبدیل پروژه ARASB به وب‌سایت واقعی، عملیاتی و توسعه‌پذیر

## 1) خلاصه مدیریتی
پروژه فعلی ARASB یک لندینگ‌پیج فرانت‌اند با React/Vite/Tailwind است که از نظر طراحی ظاهری مناسب، سبک و قابل ارائه است، اما هنوز برای استفاده واقعی تجاری، مدیریت محتوا، سئو حرفه‌ای، تولید صفحات، چندزبانگی کامل، مدیریت محصول، ارتباط با مشتری، گزارش‌گیری، امنیت، بک‌اند، پنل مدیریت و فرایند به‌روزرسانی آماده نیست.

این سند، نقشه راه کامل برای تبدیل پروژه فعلی به یک وب‌سایت شرکتی/تجاری حرفه‌ای چندزبانه با پنل مدیریت فارسی، صفحه‌ساز، مدیریت محصولات، سئوی کامل، ابزارهای تحلیلی، و سازوکار آپدیت از داخل پنل را مشخص می‌کند.

## 5.2 صفحه‌ساز و ویرایشگر صفحات

برای اینکه بدون دخالت توسعه‌دهنده بشود صفحات را ساخت و ویرایش کرد، نیاز به یک **page builder / block editor** داخلی وجود دارد.

### امکانات پیشنهادی
- ساخت صفحه با بلوک‌ها
- Drag & Drop یا حداقل Block-based editor
- بلوک‌های آماده:
  - Hero
  - متن و عنوان
  - تصویر
  - گالری
  - اسلایدر
  - ویدیو
  - FAQ
  - CTA
  - لیست ویژگی‌ها
  - آمار و اعداد
  - لوگو مشتریان
  - جدول
  - فرم
  - نقشه
  - HTML سفارشی
  - محصولات منتخب
  - مقالات منتخب
  - سکشن کشورهای مقصد
  - timeline / process
- تعیین layout برای هر بلوک
- فعال/غیرفعال‌کردن بلوک‌ها
- تنظیم فاصله‌ها، رنگ‌ها، پس‌زمینه‌ها
- نسخه مجزا برای هر زبان
- سئوی مجزا برای هر صفحه

### هشدار اجرایی
اگر page builder خیلی سنگین و پیچیده طراحی شود، پروژه زمان و هزینه زیادی می‌گیرد. پیشنهاد فازبندی:
- فاز اول: Block editor استاندارد و محدود

## 15) سیستم آپدیت از داخل پنل
کاربر اعلام کرده که نمی‌خواهد برای هر آپدیت وارد هاست شود و فایل‌ها را دستی آپلود کند. بنابراین باید یک **Update Manager** طراحی شود.

## 15.1 تعریف نیاز
- ادمین بتواند از داخل پنل، فایل آپدیت را آپلود کند.
- سیستم قبل از اعمال آپدیت، از نسخه فعلی backup بگیرد.
- فایل آپدیت اعتبارسنجی شود.
- migrationها اجرا شوند.
- build/restart انجام شود.
- در صورت شکست، rollback انجام شود.
- لاگ کامل عملیات نگهداری شود.

## 15.2 محدودیت و واقعیت اجرایی
این قابلیت بسیار حساس است و اگر درست طراحی نشود، از نظر امنیتی و پایداری خطرناک است.

### نکته مهم
"آپلود فایل آپدیت از پنل" باید محدود شود به:
- فقط مدیرکل
- فقط فایل‌های امضاشده/معتبر
- فقط بسته مشخص با ساختار استاندارد
- فقط در سرورهایی که دسترسی لازم دارند

## 15.3 روش پیشنهادی امن
### روش A: Package Update Manager
- فایل zip آپدیت با ساختار استاندارد آپلود می‌شود.
- شامل manifest.json است.
- شامل نسخه، checksum، migration، و فایل‌های لازم است.
- سیستم آن را در فضای staging استخراج می‌کند.
- صحت‌سنجی انجام می‌شود.
- backup گرفته می‌شود.
- migrationها اجرا می‌شود.
- release جایگزین می‌شود.
- سرویس reload می‌شود.

### روش B: Git-based deploy from panel
- پنل فقط trigger deploy می‌زند.
- سرور از repository یا artifact جدید pull می‌کند.
- امن‌تر و حرفه‌ای‌تر از آپلود فایل خام است.

### پیشنهاد اجرایی
برای فاز اول:
- **نسخه امن‌تر: Git/Artifact-based deployment**
- اگر الزام واقعی بر آپلود فایل باشد، در فاز دوم **Update Package Manager** طراحی شود.

## 15.4 امکانات پنل آپدیت
- نمایش نسخه جاری
- نمایش نسخه بسته آپلودی
- توضیحات آپدیت
- بررسی پیش‌نیازها
- backup now
- apply update
- rollback
- مشاهده لاگ‌ها
- هشدار ناسازگاری
