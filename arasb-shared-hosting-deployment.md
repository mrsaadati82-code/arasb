# راهنمای استقرار ARASB روی هاست لینوکسی اشتراکی

## 1) هدف سند
این سند با فرض اجرای پروژه روی **هاست لینوکسی اشتراکی** نوشته شده و مشخص می‌کند چگونه باید ساختار فنی، خروجی build و بسته‌های نصب/آپلود آماده شوند.

نکته مهم: از این به بعد، هر زمان خروجی قابل نصب لازم باشد، باید **فایل zip** آماده شود تا بتوانید در هاست آپلود کنید.

---

## 2) محدودیت‌های هاست اشتراکی
معمولاً هاست اشتراکی این محدودیت‌ها را دارد:
- دسترسی root ندارد
- Docker ندارد
- اجرای سرویس‌های دائمی محدود یا ناممکن است
- اجرای NestJS یا Node server دائمی ممکن است دشوار یا محدود باشد
- تنظیمات reverse proxy در اختیار شما نیست
- queue/worker دائمی معمولاً محدود است

بنابراین معماری نهایی باید با این واقعیت تطبیق پیدا کند.

---

## 3) نتیجه مهم معماری برای هاست اشتراکی
### سمت سایت عمومی
بهترین گزینه برای هاست اشتراکی:
- **خروجی استاتیک یا نزدیک به استاتیک**
- صفحات از قبل build شوند
- فایل‌های نهایی zip شوند و آپلود شوند

### سمت پنل مدیریت
دو سناریو وجود دارد:
1. پنل هم به صورت build قابل آپلود روی هاست باشد
2. پنل روی ساب‌دامین/مسیر جدا و با پشتیبانی Node اجرا شود

### سمت بک‌اند
برای هاست اشتراکی، اجرای یک بک‌اند Node کامل ممکن است سخت باشد. بنابراین باید قبل از شروع نهایی، یکی از این تصمیم‌ها گرفته شود:

#### گزینه A
هاست اشتراکی با Node.js app support واقعی دارد
- در این حالت می‌توان backend را محدودتر و کنترل‌شده اجرا کرد

#### گزینه B
هاست فقط PHP/static را خوب پشتیبانی می‌کند
- در این حالت معماری NestJS روی همان هاست اشتراکی مناسب نیست
- باید یا backend جای دیگری مستقر شود یا stack تغییر کند

---

## 4) توصیه عملی مهم
اگر اصرار بر هاست اشتراکی دارید، باید اول مشخص شود:
- آیا هاست شما Node.js را پشتیبانی می‌کند؟
- آیا امکان اجرای app دائمی وجود دارد؟
- آیا SSH دارید؟
- آیا cron jobs دارید؟
- آیا DB PostgreSQL دارد یا فقط MySQL؟

بدون این اطلاعات، معماری backend نهایی نمی‌تواند قطعی شود.

---

## 5) استراتژی پیشنهادی سازگار با هاست اشتراکی
### بهترین استراتژی کم‌ریسک
- **سایت عمومی:** build خروجی zip و آپلود روی هاست
- **پنل مدیریت:** بسته به امکانات هاست، build uploadable یا استقرار جدا
- **بک‌اند:** در صورت ضعف هاست اشتراکی، روی سرویس جدا یا تغییر تکنولوژی

---

## 6) از این به بعد در تحویل‌ها چه باید رعایت شود
هر زمان نیاز به نصب/تحویل اجرایی باشد:
1. فایل‌ها داخل پوشه مشخص آماده شوند
2. فایل `README` یا راهنمای نصب داشته باشند
3. در صورت نیاز، فایل `.env.example` داشته باشند
4. **فایل zip نهایی ساخته شود**
5. نام zip نسخه‌دار و واضح باشد

### فرمت پیشنهادی نام فایل‌ها
- `arasb-web-v0.1.0.zip`
- `arasb-admin-v0.1.0.zip`
- `arasb-foundation-v0.1.0.zip`

---

## 7) وضعیت فعلی
برای foundation فعلی، فایل zip ساخته شده است:
- `arasb-next-foundation/arasb-next-foundation-shared-hosting-foundation.zip`

این فایل فعلاً یک اسکلت پایه است، نه نسخه نهایی قابل لانچ.

---

## 8) جمع‌بندی
من از این به بعد در هر مرحله‌ای که خروجی اجرایی لازم باشد، فایل zip هم آماده می‌کنم. همچنین مسیر توسعه را با درنظرگرفتن محدودیت‌های هاست لینوکسی اشتراکی تنظیم می‌کنم؛ اما برای تصمیم نهایی backend، باید مشخصات هاست شما دقیق‌تر روشن شود.
